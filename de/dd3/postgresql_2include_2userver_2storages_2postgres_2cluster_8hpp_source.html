21 /// @page pg_topology uPg: Cluster topology discovery
22 ///
23 /// @par Principles of PgaaS role determination
24 /// - Every host except master is in recovery state from PostgreSQL's POV.
25 /// This means the check 'select pg_is_in_recovery()' returns `false` for the
26 /// master and `true` for every other host type.
27 /// - Some hosts are in sync slave mode. This may be determined by executing
28 /// 'show synchronous_standby_names' on the master.
29 /// See
30 /// https://www.postgresql.org/docs/current/runtime-config-replication.html#GUC-SYNCHRONOUS-STANDBY-NAMES
31 /// for more information.
32 ///
33 /// @par PgaaS sync slaves lag
34 /// By default, PgaaS synchronous slaves are working with 'synchronous_commit'
35 /// set to 'remote_apply'. Therefore, sync slave may be lagging behind the
36 /// master and thus is not truly 'synchronous' from the reader's POV,
37 /// but things may change with time.
38 ///
39 /// @par Implementation
40 /// Topology update runs every second.
41 ///
42 /// Every host is assighed a connection with special ID (4100200300).
43 /// Using this connection we check for host availability, writability
44 /// (master detection) and perform RTT measurements.
45 ///
46 /// After the initial check we know about master presence and RTT for each host.
47 /// Master host is queried about synchronous replication status. We use this
48 /// info to identify synchronous slaves and to detect "quorum commit" presence.
